# InfluxDB No-Code 查询平台 · 产品作品集

> 以产品经理视角重新梳理 ATSYS InfluxDB No-Code 查询与可视化平台——
> 一个让非技术用户无需编写任何 Flux 代码、即可自主完成数据探索与图表生成的 B 端工具。

## 原项目背景

InfluxDB 是工业级时序数据库，广泛用于 IoT 与运营监控场景。其原生查询语言 Flux 门槛高，导致工厂管理人员、业务分析师等非技术用户无法自主获取数据，每次数据需求都需依赖工程师介入，响应周期长、效率低。
本项目目标：为非技术用户设计零代码数据查询与可视化平台，使其无需了解任何查询语法，即可通过拖拽与下拉操作完成数据探索，结果以 Grafana 图表形式实时呈现。

## 核心产品决策

**决策1：下拉拖拽 vs 自然语言输入**
早期讨论过让用户直接输入自然语言查询。最终选择下拉拖拽，核心原因是目标用户（工厂管理人员）对 AI 黑盒结果有可控性顾虑，需要清楚知道自己"选了什么"。自然语言查询作为 P2 AI 升级路径保留，并在 Demo 的 AI 模式中实现。

**决策2：Flux 代码可选可见**
客户在 Sprint 2 明确要求「保持界面简洁，非程序员为核心用户，Flux 代码作为可选功能」。据此设计了「View Flux」按钮——高级用户可查看和调试，普通用户完全感知不到代码。

**决策3：单次登录自动连接双系统**
用户只需输入一次 InfluxDB Token，系统后台自动完成 InfluxDB 与 Grafana 双平台鉴权，用户无需分别配置——降低非技术用户的操作门槛。

**决策4：Data Tree 与下拉选择配合使用**
Data Tree 可视化展示数据层级结构，用户可跨层级拖拽任意节点至过滤区，满足需要先探索数据结构的用户；下拉选择适合目标明确的用户。两者配合，覆盖不同使用场景。

## 产品转折点

Sprint 4 客户反馈指出，团队过于专注后端实现，忽视了前端直觉体验。这次反馈成为项目转折点，团队重新聚焦用户界面与可视化设计。这让我深刻理解：技术实现如果不能转化为可感知的用户体验，产品价值会大打折扣。

## PM 视角升级：AI 模式

原项目实现了完整的 No-Code 查询构建器，但用户仍需手动完成多个下拉选择步骤。作为产品作品集的一部分，我个人设计并实现了 **AI 自然语言查询模式**，原因有三：

1. **呼应原始需求**：Sprint 2 客户曾提出「是否支持简单与高级两种查询模式」，团队当时因时间限制未能实现。AI 模式正是对这一未竟需求的回应。
2. **体现 AI PM 思维**：将 LLM 引入产品，不是为了追热点，而是解决真实痛点——用户描述「我想看过去 24 小时温度趋势」，比让他逐步选择 Bucket、Measurement、Field 更自然。
3. **保持设计一致性**：AI 模式保留了 Flux 语句可见的设计，用户可以看到 AI 解析出的参数和生成的查询语句，与原项目「透明度优先」的理念一致，而不是黑盒。
